### EAV Schema (Entity-Attribute-Value)

DenchClaw stores CRM data in a flexible EAV pattern. The core tables (exact columns from the real schema):

**`objects`** — no `label` column, no `icon` column (icon lives in .object.yaml only):
```sql
objects (id, name, description, default_view, parent_document_id, sort_order, source_app, immutable, created_at, updated_at)
```

**`fields`** — `name` is the display name ("Company Name" not "company_name"), `required` not `is_required`:
```sql
fields (id, object_id, name, description, type, required, default_value, related_object_id, relationship_type, enum_values JSON, enum_colors JSON, enum_multiple, sort_order, created_at, updated_at)
-- UNIQUE(object_id, name)
```

**`statuses`** — no `label` column, has `is_default`:
```sql
statuses (id, object_id, name, color, sort_order, is_default, created_at, updated_at)
-- UNIQUE(object_id, name)
```

**`entries`** and **`entry_fields`**:
```sql
entries (id, object_id, sort_order, created_at, updated_at)
entry_fields (id, entry_id, field_id, value VARCHAR, created_at, updated_at)
-- UNIQUE(entry_id, field_id)
```

### Naming Conventions (from real CRM skill)

- **Object names**: singular, lowercase, one word ("account" not "Accounts")
- **Field names**: human-readable, proper capitalization ("Company Name" not "company_name", "Email Address" not "email")
- **PIVOT view column names**: match field names exactly — use double-quotes in SQL for names with spaces: `"Company Name"`

### PIVOT Views

Use DuckDB's native `PIVOT` syntax (NOT the MAX(CASE WHEN) pattern):

```sql
CREATE OR REPLACE VIEW v_account AS
PIVOT (
  SELECT e.id as entry_id, e.created_at, e.updated_at,
         f.name as field_name, ef.value
  FROM entries e
  JOIN entry_fields ef ON ef.entry_id = e.id
  JOIN fields f ON f.id = ef.field_id
  WHERE e.object_id = (SELECT id FROM objects WHERE name = 'account')
    AND f.type != 'action'
) ON field_name IN ('Company Name', 'Domain', 'Industry', 'Employee Count',
  'Annual Revenue', 'HQ City', 'HQ Country', 'Owner', 'Phone', 'Website',
  'Description') USING first(value);
```

**CRITICAL:** Always list field names explicitly in the `IN (...)` clause. Query PIVOT views with quoted field names: `SELECT "Company Name" FROM v_account WHERE "Industry" = 'Manufacturing'`

### Object Triple Identity

1. **DuckDB row** — `objects.name = 'account'`
2. **Filesystem directory** — `{{WORKSPACE_PATH}}/account/`
3. **YAML metadata** — `{{WORKSPACE_PATH}}/account/.object.yaml` with `name: "account"`

Workspace path is `~/.openclaw-dench/workspace/` (user-local, NOT in the repo).

### Field Types

`text`, `email`, `phone`, `url`, `number`, `boolean`, `date`, `richtext`, `file`, `user`, `enum`, `relation`, `tags`, `action`

- `enum` fields: store options in `enum_values JSON` column, colors in `enum_colors JSON`
- `relation` fields: `related_object_id` points to target object, `relationship_type` is the cardinality
- `action` fields: excluded from PIVOT views (`AND f.type != 'action'`)
- All values stored as VARCHAR in `entry_fields.value` — cast in queries (e.g., `::NUMERIC`, `::DATE`)

### Plugin Structure

Extensions live in `extensions/<name>/` with flat file layout (NO deep `src/` trees):

Every extension MUST have these three files:
- **`package.json`** — with `"openclaw": { "extensions": ["./index.ts"] }`
- **`openclaw.plugin.json`** — manifest with `id`, `name`, `version`, `configSchema`
- **`index.ts`** — `export default function register(api)` + `export const id = "..."`

Additional `.ts` files sit flat alongside index.ts (e.g., `sync.ts`, `search.ts`, `crdt.ts`).
Test files sit alongside code: `index.test.ts`, `sync.test.ts`, etc.

## 5. Stack-Specific Rules

### TypeScript
- Strict mode — no `any`, no implicit any, no type assertions unless truly necessary
- All function signatures typed (input and return)
- Use `interface` for object shapes, `type` for unions/intersections
- Prefer `const` over `let`, never use `var`

### DuckDB
- All queries via parameterized prepared statements
- Never use string interpolation for values — only for known table/column names
- Single-writer constraint: user writes go direct, sync writes go through `sync_queue` table
- Use transactions for multi-statement operations
- PIVOT views are read-only — write through the EAV tables

### Plugin Tools
- Import types: `import type { AnyAgentTool, OpenClawPluginApi } from "openclaw/plugin-sdk"`
- Export pattern: `export const id = "b2b-crm"` + `export default function register(api: OpenClawPluginApi)`
- Every tool registered via `api.registerTool()` must have:
  - `name`: snake_case string, prefixed `b2b_crm_*`
  - `label`: human-readable display name
  - `description`: clear sentence describing what the tool does
  - `parameters`: JSON Schema object defining inputs
  - `execute`: `(toolCallId: string, params: Record<string, unknown>) => ToolExecutionResult`
- Tool return type: `{ content: [{ type: "text" as const, text: "..." }], details?: unknown }`
- Use `as AnyAgentTool` cast on tool objects

### Skills (SKILL.md)
- YAML frontmatter: `name`, `description`, `metadata: { "openclaw": { "inject": true, "always": true } }`
- Body is natural language instructions teaching the agent SQL patterns
- Include example queries the agent should run — use `"Quoted Field Names"` in all SQL
- Never put executable code, imports, or function definitions in skills
- Reference PIVOT view names explicitly (e.g., "query v_account for account data")
- Use `{{WORKSPACE_PATH}}` template variable for workspace directory paths

### Tests (vitest)
- Test files sit alongside source: `extensions/b2b-crm/*.test.ts` (NOT in a `tests/` subdirectory)
- Use `describe` / `it` blocks
- Mock the plugin API with a `createApi()` helper (see apollo-enrichment test pattern)
- Test real DuckDB operations against in-memory database where needed
- Mock only external services (cloud sync target)
