## GOAL — System Invariants

**Core flows that must never break:**
1. Create an account, contact, and deal — all appear in DenchClaw web UI via PIVOT views
2. Associate contacts with deals via roles (champion, decision-maker, blocker)
3. Move deals through pipeline stages — transition history logged automatically
4. Sync local changes to mock cloud and pull remote changes back — CRDT merge resolves conflicts
5. Search 10K+ accounts with full-text, boolean filters, and faceted counts
6. Import CSV with adversarial data — bad rows skip-and-log, good rows import
7. Activity events captured on CRUD operations — engagement scores and neglect flags queryable

**Architecture constraints (do not change without STRUCTURAL CHANGE REQUIRED):**
- CRM objects (account, contact, deal) are EAV-native — rows in `objects`, `fields`, `entries`, `entry_fields`
- Internal tables (line_items, sync_queue, activity_events, etc.) are standalone DuckDB tables
- All sync writes go through `sync_queue` — never write directly to DuckDB from sync tools
- Every EAV object has triple identity: DuckDB row + filesystem directory + .object.yaml
- Skills contain only markdown + YAML frontmatter — never executable code
- Plugin tools are the only executable extension surface

### PIVOT Views

```sql
-- Use DuckDB native PIVOT syntax. CRITICAL: always list field names in IN (...) clause.
-- Query non-action fields first to build the list:
-- SELECT name FROM fields WHERE object_id = (SELECT id FROM objects WHERE name = 'account') AND type != 'action' ORDER BY sort_order;

CREATE OR REPLACE VIEW v_account AS
PIVOT (
  SELECT e.id as entry_id, e.created_at, e.updated_at,
         f.name as field_name, ef.value
  FROM entries e
  JOIN entry_fields ef ON ef.entry_id = e.id
  JOIN fields f ON f.id = ef.field_id
  WHERE e.object_id = (SELECT id FROM objects WHERE name = 'account')
    AND f.type != 'action'
) ON field_name IN ('Company Name', 'Domain', 'Industry', 'Employee Count',
  'Annual Revenue', 'HQ City', 'HQ Country', 'Owner', 'Phone', 'Website',
  'Description', 'Tags') USING first(value);

-- v_contact and v_deal follow the same PIVOT pattern with their respective field names.
-- Query columns with quotes: SELECT "Company Name" FROM v_account WHERE "Industry" = 'Manufacturing'
```

```typescript
import { Database, Connection } from 'duckdb-async';

let db: Database | null = null;

export async function getDb(dbPath: string = 'workspace.duckdb'): Promise<Database> {
  if (!db) {
    db = await Database.create(dbPath);
  }
  return db;
}

export async function getConnection(dbPath?: string): Promise<Connection> {
  const database = await getDb(dbPath);
  return database.connect();
}

export async function runQuery<T = Record<string, unknown>>(
  sql: string,
  params: unknown[] = [],
  dbPath?: string
): Promise<T[]> {
  const conn = await getConnection(dbPath);
  try {
    return await conn.all(sql, ...params) as T[];
  } finally {
    await conn.close();
  }
}

export async function execQuery(
  sql: string,
  params: unknown[] = [],
  dbPath?: string
): Promise<void> {
  const conn = await getConnection(dbPath);
  try {
    await conn.exec(sql, ...params);
  } finally {
    await conn.close();
  }
}

export async function closeDb(): Promise<void> {
  if (db) {
    await db.close();
    db = null;
  }
}
```

**0.5 — Write migration runner skeleton (migrations.ts)**
Write `extensions/b2b-crm/migrations.ts`:
```typescript
import { execQuery, runQuery } from './db.js';

interface Migration {
  version: number;
  name: string;
  up: string;
}

export async function runMigrations(migrations: Migration[]): Promise<void> {
  await execQuery(`
    CREATE TABLE IF NOT EXISTS b2b_crm_migrations (
      version INTEGER PRIMARY KEY,
      name VARCHAR NOT NULL,
      applied_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    )
  `);

  const applied = await runQuery<{ version: number }>(
    'SELECT version FROM b2b_crm_migrations ORDER BY version'
  );
  const appliedVersions = new Set(applied.map(r => r.version));

  for (const migration of migrations.sort((a, b) => a.version - b.version)) {
    if (appliedVersions.has(migration.version)) continue;

    await execQuery(migration.up);
    await execQuery(
      'INSERT INTO b2b_crm_migrations (version, name) VALUES (?, ?)',
      [migration.version, migration.name]
    );
  }
}
```

### PHASE 2 — Sync Protocol
**Status:** `[ ] Not started`
**Goal:** Bidirectional sync between local DuckDB and mock cloud (second DuckDB). HLC-based LWW-Register CRDTs resolve field-level conflicts deterministically. Sync queue decouples sync writes from user writes.
**Acceptance:** Two nodes make conflicting edits to the same field offline → after sync, both converge to the same value (higher HLC wins). Two nodes edit different fields on the same record → both values preserved (auto-merge). `sync.test.ts` passes.

**Before you start:** State your plan in 3–5 bullets.

#### Tasks

**2.1 — Hybrid Logical Clock (hlc.ts)**
Write `extensions/b2b-crm/hlc.ts`:
```typescript
export interface HLC {
  ts: number;      // physical timestamp in ms (Date.now())
  counter: number; // tie-breaker for same-ms events
  nodeId: string;  // unique identifier for this node
}

export function createHLC(nodeId: string): HLC
export function incrementHLC(clock: HLC): HLC
export function receiveHLC(local: HLC, remote: HLC): HLC
export function compareHLC(a: HLC, b: HLC): number  // -1, 0, 1
export function serializeHLC(hlc: HLC): string
export function deserializeHLC(s: string): HLC
```
- `incrementHLC`: set `ts = max(clock.ts, Date.now())`, increment counter if ts unchanged, reset to 0 if ts advanced
- `receiveHLC`: set `ts = max(local.ts, remote.ts, Date.now())`, set counter = max of relevant counters + 1
- `compareHLC`: compare ts first, then counter, then nodeId (lexicographic tiebreak)

```sql
-- Recency: days since last activity, inverted and normalized (0-1)
-- Frequency: count of events in window, log-scaled
-- Depth: weighted sum of event types (create=3, update=2, view=1, search=0.5)
-- Score = 0.4 * recency + 0.3 * frequency + 0.3 * depth
SELECT
  entity_id,
  entity_type,
  (1.0 / (1.0 + DATEDIFF('day', MAX(occurred_at), CURRENT_TIMESTAMP))) AS recency,
  LN(1 + COUNT(*)) / LN(1 + max_count) AS frequency,
  SUM(CASE event_type
    WHEN 'create' THEN 3 WHEN 'update' THEN 2 WHEN 'view' THEN 1
    WHEN 'stage_change' THEN 2.5 WHEN 'search' THEN 0.5 ELSE 1 END
  ) / max_depth AS depth,
  MAX(occurred_at) AS last_activity
FROM activity_events
WHERE occurred_at >= CURRENT_TIMESTAMP - INTERVAL ? DAY
GROUP BY entity_id, entity_type
```

```sql
  SELECT entity_id, DATE_TRUNC('day', occurred_at) AS day, COUNT(*) AS cnt
  FROM activity_events

  GROUP BY entity_id, DATE_TRUNC('day', occurred_at)

  SELECT entity_id,
    AVG(cnt) OVER (PARTITION BY entity_id ORDER BY day ROWS BETWEEN 29 PRECEDING AND CURRENT ROW) AS avg_cnt,
    STDDEV(cnt) OVER (PARTITION BY entity_id ORDER BY day ROWS BETWEEN 29 PRECEDING AND CURRENT ROW) AS std_cnt,
    cnt AS current_cnt, day
  FROM daily_counts
```
